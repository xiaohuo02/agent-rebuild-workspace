# 01-整体架构设计

## 架构定位

本工作区采用 Agentic Workflow（智能体工作流）架构。

核心思想：

```text
用户问题
  -> FastAPI 接入
  -> Router 意图识别
  -> Guardrails 安全护栏
  -> Planner 任务规划
  -> Tool Selection 工具选择
  -> Tool Execution 工具执行
  -> Evidence 证据链
  -> Final Answer 最终回答
```

## 为什么这样设计

客服系统不能直接把用户问题交给 LLM（大语言模型）自由回答，因为它会遇到三个风险：

1. 事实风险：价格、库存、售后政策必须来自真实数据。
2. 权限风险：用户不能越权查询别人订单。
3. 动作风险：退款、改地址、补偿等动作必须受控。

所以模型只负责理解、规划和表达，工具负责查真实数据，工作流负责控制边界。

## 第一阶段分层

| 层 | 作用 | 第一阶段实现 |
|---|---|---|
| 接入层 | 承接 HTTP 请求 | FastAPI |
| 前端层 | 对话式体验 | Vue 3 + Vite |
| 编排层 | 控制 Agent 流程 | 先用 Python 服务编排，后续替换 LangGraph |
| 工具层 | 查询业务事实 | fake 商品工具、fake 售后政策工具 |
| 证据层 | 约束最终回答 | Evidence 结构 |
| 测试层 | 验证主链路 | pytest + eval dataset 骨架 |

## 为什么第一阶段不直接接 Neo4j 和 GraphRAG

第一阶段的目标不是功能堆满，而是先证明架构链路成立。

用 fake data（假数据）有三个好处：

1. 不依赖数据库环境，启动成本低。
2. 结果可控，便于写测试。
3. 更容易讲清 Agent 每个节点的职责。

后续替换路径：

```text
fake product tool -> Neo4j + Text2Cypher
fake policy tool  -> GraphRAG
fake LLM          -> DeepSeek / Ollama
```

## 前端形态

前端采用 ChatGPT 风格对话工作台：

```text
左侧：会话列表
中间：消息流和输入框
右侧：Agent 过程和证据链
```

右侧面板用于学习和面试演示，展示 Router（路由器）、Planner（规划器）、Tool Calling（工具调用）和 Evidence（证据链）。

## 面试讲法

> 我把项目设计成 Agentic Workflow（智能体工作流），不是让大模型自由回答。入口层负责协议和流式返回，Agent 编排层负责路由、护栏、规划和工具选择，工具层负责查事实数据，Evidence（证据链）约束最终回答。第一阶段先用假数据跑通闭环，后续再替换成 Neo4j、Text2Cypher 和 GraphRAG。
